# Project Chimera: Game Design Document

## The RTS Creation Platform — Complete Production Specification

**Project Chimera is not an RTS game. It is the platform that ships the RTS genre as a living, community-owned system.** Players divide into three roles: Commanders who play scenarios, Architects who build them, and Tinkerers who do both. Every design decision in this document answers one of three questions: Does this make it easier to *create* a great RTS? Easier to *share* one? More exciting to *discover and play* one made by someone else?

This document defines the full technical and design specification for a solo-developed, AI-assisted RTS creation platform built on Godot 4.6 with C#. It covers core gameplay systems, the creation suite, multiplayer architecture, the UGC pipeline, AI asset generation, and a phased build plan spanning 19–31 months. Project Chimera launches as a compelling single-player RTS, then grows into a creation platform through Early Access iteration.

---

## 1. Vision and identity

### The three-question filter

Every feature, system, and line of code must serve at least one of these questions:

- **Create**: Does this make it easier to build a great RTS scenario?
- **Share**: Does this make it easier to publish and distribute that scenario?
- **Discover**: Does this make it more exciting to find and play scenarios made by others?

Features that serve none of these get cut. Features that serve two or three get prioritized.

### Player archetypes

**Commanders** are pure players. They want to discover scenarios, play against AI or humans, compete, and experience stories. They never open the editor. They represent the majority of the player base — Fortnite Creative data shows only ~0.2% of daily active users create content. Commanders are the audience that makes creation meaningful.

**Architects** are builders. They paint terrain, place units, define factions, wire triggers, and publish scenarios. They may rarely play competitively. They are the lifeblood of the platform. The Warcraft III World Editor proved that accessible creation tools can birth entire genres — DotA, Tower Defense, and dozens of custom game modes emerged from a modding community empowered by approachable tools.

**Tinkerers** bridge both worlds. They play extensively, then build scenarios informed by that play experience. They push the creation tools to their limits, use visual scripting, tweak balance, and iterate through rapid play-test cycles. Tinkerers produce the highest-quality content because they understand both sides.

### Design pillars

**Data-driven everything.** No game logic is hardcoded. Units, buildings, resources, tech trees, win conditions, combat rules — all defined as data that creators modify without writing code. The framework processes data; creators supply it.

**Layered complexity.** Every system exposes a simple mode (presets, dropdowns, wizards) and an advanced mode (data editing, visual scripting, raw JSON). This mirrors the Warcraft III World Editor's success: GUI triggers for beginners, JASS scripting for experts. Progressive disclosure prevents overwhelm while preserving depth.

**Composition over inheritance.** A "healer" is not a special unit class — it is a ranged unit plus a heal ability plus a support AI behavior. All complexity emerges from combining simple, orthogonal components.

### Art direction

Project Chimera adopts a **stylized low-poly aesthetic** with clean geometric shapes, soft colors, and slight cel-shading. This choice is strategic: low-poly assets are the most reliably generated by current AI 3D tools (Hunyuan3D, Tripo AI), maintain visual consistency across hundreds of generated assets, render efficiently at RTS unit counts, and read clearly at the zoomed-out camera distances RTS players use. The style sits between Mindustry's utilitarian clarity and Northgard's painterly warmth.

## 2. Technical foundation

### Engine selection: Godot 4.6.2 stable

Project Chimera targets **Godot 4.6.2**, released April 1, 2026. This is the current stable version as of this document's writing. Godot 4.7 is in active development but will not be used for production until it reaches stable status.

Godot 4.6 delivers several features critical to this project. **Jolt Physics** is now the default engine, handling approximately **800 CharacterBody3D instances** before degradation — a 16x improvement over the previous GodotPhysics engine. The **GDScript JIT compiler** provides 3–8x speedups, though C# remains 2–3x faster for the computational workloads this project demands. **LibGodot** enables the engine to run as an embeddable library, opening doors for custom tool hosts. **Patch PCK delta encoding** reduces update sizes for deployed projects, directly benefiting UGC distribution.

### Language: C# with .NET 9

C# via GodotSharp is **production-ready for desktop targets** in Godot 4.6. The .NET 9 runtime provides AOT compilation improvements and access to the full NuGet package ecosystem. Benchmarks from the StraySpark 2026 evaluation show C# processing 1,000 AI agent state machines in **0.7ms** versus GDScript JIT's 1.9ms — a critical advantage for RTS simulation loops processing hundreds of entities per frame.

Known limitations accepted for this project: no C# web export (irrelevant for a PC-targeted RTS), no direct C# calls to GDExtensions (workaround via GDScript bridges where needed), and ~30–60MB additional export size for the .NET runtime. Only **16% of Godot developers** use C#, meaning fewer community tutorials, but the tradeoff is justified by performance requirements and access to NuGet libraries for networking, serialization, and fixed-point math.

The Chickensoft community provides essential C# tooling: source generators for scene tree bindings, input mapping code generation, and GdUnit4 for unit testing.

### Architecture: ECS-inspired simulation with Godot scene rendering

The core architecture separates **simulation** from **presentation**. The simulation layer uses plain C# structs in arrays (Struct of Arrays pattern) for all entity data — positions, health, combat states, pathfinding targets. No Godot Nodes exist per unit in the simulation layer. This allows SIMD-friendly memory layouts, parallel processing via `Parallel.For`, and cache-coherent iteration over thousands of entities.

The presentation layer uses **MultiMeshInstance3D** for unit rendering. Each visual unit type (infantry, archer, siege engine) maps to one MultiMesh, rendering all instances in a single draw call. A typical RTS match with 5 unit types across 2 factions requires only 10 draw calls for all units regardless of count. Transform data flows from the simulation arrays to MultiMesh instance transforms each frame.

```
Simulation Layer (Pure C#)           Presentation Layer (Godot Nodes)
┌─────────────────────────┐         ┌───────────────────────────┐
│ Entity[] positions       │ ──────→ │ MultiMeshInstance3D (per  │
│ Entity[] health          │         │ unit type, 1 draw call)   │
│ Entity[] combatState     │         ├───────────────────────────┤
│ Entity[] pathTarget      │         │ Terrain3D (GDExtension)   │
│ Entity[] factionId       │         ├───────────────────────────┤
│                          │         │ FogOfWar viewport overlay │
│ FixedPoint math          │         ├───────────────────────────┤
│ Deterministic RNG        │         │ UI (Control nodes)        │
│ NavigationServer queries │         └───────────────────────────┘
└─────────────────────────┘
```

This architecture targets **500–2,000 units at 60 FPS** with C# simulation. If profiling reveals bottlenecks beyond C# capability, the simulation hot path can migrate to a GDExtension (C++) without changing the architecture — only the execution layer changes.

### Pathfinding architecture

NavigationServer3D serves as the foundation, but direct API calls replace NavigationAgent3D nodes. Path queries use `NavigationServer3D.MapGetPath()` called from C# with aggressive rate-limiting: a maximum of **30 path calculations per physics frame**, with each unit recalculating its path only every 300ms or when its command changes. Community testing confirms this approach handles **2,000 agents at 140 FPS** on an i5-12600k.

RVO (Reciprocal Velocity Obstacles) avoidance is disabled for most units — the official Godot documentation warns that "avoidance processing with many registered agents has a significant performance cost." Instead, units use lightweight steering behaviors: separation forces between nearby allies, simple collision radius checks via a spatial hash grid, and the "harvester hack" (disabling inter-unit collision for gatherers in transit, as StarCraft famously did).

For groups of 4+ units sharing a destination, the system switches to **flow fields** — a single vector field computation serves all units moving to the same target. No production-ready Godot flow field implementation exists, so this requires a custom C# implementation. The architecture uses hierarchical pathfinding: coarse region-level A* for long-range routing, flow fields for medium-range group movement, and local steering for close-range avoidance.

### Dedicated server pipeline

Godot 4.6 supports dedicated server exports natively. The project uses a **single-codebase architecture** with feature tags to branch client and server code. Running any Godot export with `--headless` produces a server build; the dedicated server export mode additionally strips textures, audio, and visual resources to minimize the server binary.

Server detection in code: `OS.HasFeature("dedicated_server")` or `DisplayServer.GetName() == "headless"`. The server exports target **Linux** and deploy via Docker containers. The C# build uses NativeAOT for self-contained deployment without requiring the .NET runtime on the server machine.

### Runtime editor capability assessment

Godot 4's runtime introspection is a **core engine strength** and the primary reason this creation-platform concept is viable. At runtime, the game can enumerate all properties of any object via `GetPropertyList()`, read and write them via `Get()`/`Set()`, dynamically instantiate and rearrange scene trees via `AddChild()`/`RemoveChild()`, and pack runtime scenes back into `PackedScene` resources via `PackedScene.Pack()`. The Control node UI system — the same system that powers the Godot editor itself — is fully available at runtime, including `GraphEdit` and `GraphNode` for visual scripting interfaces.

The proven pattern (validated by projects like Gravity Ace): build property inspectors by iterating `GetPropertyList()`, generate UI controls dynamically based on property types, save levels by serializing entity positions and properties, and toggle between PLAY and EDIT modes via a global state flag.

## 3. Core gameplay systems

All systems described below are **data-driven** and **creator-extensible**. The framework ships with sensible defaults that produce a satisfying StarCraft-like RTS out of the box. Creators override any parameter to build games ranging from classic base-builders to tower defense, survival, hero arenas, or entirely novel RTS variants.

### Resource system specification

The resource system supports **N configurable resources** with per-resource collection models. This single architecture reproduces every canonical RTS economy: StarCraft's dual mineral/gas, Age of Empires' four-resource system, Total Annihilation's streaming income model, and Warcraft III's upkeep penalties. Creators define the number, behavior, and rules of their economy.

Each resource is defined as a data object:

| Field | Type | Description | Default |
|-------|------|-------------|---------|
| `id` | string | Unique identifier | — |
| `display_name` | string | Shown in UI | — |
| `icon` | Texture | Resource icon | — |
| `collection_model` | enum | GATHER, INCOME, or STREAMING | GATHER |
| `is_finite` | bool | Does the node deplete? | true |
| `max_storage` | float | Per-player storage cap (-1 = unlimited) | -1 |
| `starting_amount` | float | Each player begins with this | 500 |

Resource nodes on the map are separate definitions with `total_supply`, `gather_rate`, `max_gatherers`, and an optional `requires_structure` flag (for refinery-on-geyser patterns).

The default configuration ships **two resources** — a primary "Ore" (abundant, funds basic units) and a secondary "Crystal" (scarce, funds advanced tech) — plus a **supply cap** system requiring infrastructure buildings. This dual-resource-plus-supply pattern is the sweet spot identified across RTS history: it forces meaningful economic decisions (which resource to prioritize, when to invest in supply buildings) without overwhelming creators or players with four or more resource types.

An optional **upkeep system** can be enabled per-scenario, applying income multipliers at configurable population thresholds. This reproduces Warcraft III's gold tax mechanic for creators who want it, without imposing it on those who don't.

### Combat system specification

Combat uses a **damage type / armor type matrix** — the most expressive and creator-friendly model available. Each unit has a damage type (Normal, Pierce, Siege, Magic, Hero) and an armor type (Unarmored, Light, Medium, Heavy, Fortified, Hero). The matrix stores float multipliers (0.25 to 3.0) defining how each damage type performs against each armor type.

Damage resolution per attack: `final_damage = base_damage × matrix[attacker.damage_type][target.armor_type] − target.armor_value`

The default matrix ships with **soft counters** (0.7–1.3 multipliers) suitable for casual play. Creators can define hard counters (0.25–3.0) for competitive depth. Beyond the matrix, each unit defines:

- **Attack cooldown** (not DPS) — this creates the burst-vs-sustain distinction that makes combat interesting
- **Splash radius** — 0 for single-target, positive values for area damage
- **Projectile vs. hitscan** flag — projectiles can miss moving targets, adding micro potential
- **Attack range** — minimum distance to engage

Combat feedback is handled by a per-unit-type `CombatFeedbackProfile` defining hit particles, impact sounds, screen shake intensity and duration, hit freeze frames, death effects, and sounds. The framework ships a default "satisfying" profile that creators override per unit. This system ensures every RTS built on Chimera has baseline "juice" — the immediate visual and audio response to combat actions that makes games feel responsive.

### Unit framework specification

Six foundational archetypes define the minimum framework. All other unit types emerge from combining stats, abilities, and behaviors:

**Worker** — gathers resources, constructs buildings. Defined by `gather_rate`, `build_speed`, `carry_capacity`. **Melee** — close-range damage dealer. Short attack range, typically higher HP. **Ranged** — attacks from distance, typically fragile. **Siege** — slow, powerful, anti-structure. High damage against Fortified armor. **Air** — exists on a separate movement layer, only targetable by units with `can_target_air` flag. **Structure** — static, produces units or research. Has a footprint, construction method, and production queue.

Additional archetypes (Support/Healer, Hero, Transport, Scout) are not separate framework classes — they're compositions of the base six with specific abilities attached. A "Healer" is a ranged unit with a Heal ability and Support AI behavior. A "Hero" is a melee or ranged unit with an XP component, leveling table, and hero abilities.

Each unit is defined by a `UnitDefinition` resource containing: display name, portrait, 3D model reference, category, all stat values (HP, attack, range, speed, armor, costs, build time), damage type, armor type, ability list, tag list (organic, mechanical, magical), combat feedback profile, and the building that produces it plus any tech prerequisites.

### Movement and command system

The framework supports these non-negotiable commands: **Move**, **Attack-Move**, **Patrol**, **Stop**, **Hold Position**, **Follow**, and **Rally Point** (per building). These represent the complete vocabulary that every RTS player expects. Creators can disable specific commands per unit type but cannot remove them from the framework.

Formation movement is configurable per unit type. Groups larger than 4 units sharing a destination use flow fields. Units in formation maintain relative positions with configurable shapes (line, box, wedge). The system implements priority-based yielding: moving units take priority over idle units, and idle units shift aside. The "harvester hack" — disabling collision between gatherers in transit — is a framework-level option per movement state, preventing the harvester traffic jams that plagued early RTS games.

Creator-customizable movement parameters include: collision radius, movement speed, speed modifiers per terrain type, whether units push or yield to each other, and formation shapes.

### Fog of war specification

The fog of war system uses a **render-texture approach** operating on a 2D grid overlaid on the game world. Each cell exists in one of three states: **Unexplored** (black — never seen, nothing rendered), **Explored** (dim — previously seen, terrain and last-known building positions visible, enemy units hidden), and **Visible** (clear — within a friendly unit's vision radius, everything rendered live).

Each simulation tick: reset all Visible cells to Explored, then for each friendly unit stamp a circle of radius `unit.vision_range` as Visible, then upload the modified texture to the GPU where a shader applies the visual overlay. An optional height-advantage toggle gives elevated terrain bonus vision range.

For multiplayer, fog of war is **server-enforced**. The server only transmits entity data for units within a player's visible cells. This prevents map hacking at the protocol level — clients literally never receive data about units they cannot see.

Creators can disable fog entirely, toggle shroud (explored vs. unexplored distinction), set per-unit vision ranges, configure fog color, and enable or disable height advantage. The system is opt-in — sandbox and cooperative scenarios can run without fog.

### Win condition and scenario logic system

Win conditions use a **tiered architecture** that scales from zero-configuration presets to full visual scripting:

**Tier 1 — Presets (zero-code).** Dropdown selector with built-in conditions: Annihilation (destroy all enemy structures and units), Landmark Destruction (destroy specific key buildings), Score-based (highest score when timer expires), King of the Hill (control a point for cumulative time), Assassination (kill a specific hero unit), and Timed Survival. Each preset exposes configurable parameters.

**Tier 2 — Trigger Editor (low-code, Warcraft III-style).** Event → Condition → Action lists with dropdown-driven sentence-builder UI. The vocabulary includes events (Unit Dies, Building Completed, Timer Expires, Region Entered, Resource Threshold Reached), conditions (Player Has X Units, Building Exists, Variable Comparison), and actions (Spawn Unit, Display Message, Set Variable, Victory/Defeat, Create Timer, Play Sound, Move Camera). This tier handles 90% of custom scenario needs.

**Tier 3 — Visual Node Graph (medium-code).** For advanced scenarios requiring branching, loops, and state machines. Built on Godot's `GraphEdit` control node at runtime. Pre-built node libraries for events, conditions, actions, and flow control. Both Tier 2 and Tier 3 edit the same underlying data model — switching between views is seamless.

**Tier 4 — Natural language input (AI-assisted).** An LLM translates plain English descriptions into Tier 2 trigger JSON. The creator types "When the player captures the northern outpost, spawn 20 enemy soldiers from the eastern forest," and the system generates, validates, and presents the structured trigger for review. This is an accelerator, never a replacement — output always passes through the validation layer and appears in the Trigger Editor for human confirmation before activation.

## 4. The Scenario Director

The Scenario Director is the runtime system that evaluates triggers, manages game flow, and executes scripted events during gameplay. It serves all three player types: Commanders experience its output as narrative beats and win conditions, Architects author it through the creation suite, and Tinkerers exploit its full depth for complex custom game modes.

### Trigger execution engine

Triggers are evaluated each simulation tick against the current game state. Each trigger has an enabled flag, a run-once flag, and an optional cooldown. The execution order is deterministic: triggers are sorted by priority (creator-assigned), then by creation order. This determinism is essential for multiplayer synchronization.

The trigger data model in JSON:

```json
{
  "name": "Enemy Reinforcements",
  "enabled": true,
  "run_once": false,
  "cooldown_seconds": 60,
  "events": [
    { "type": "timer_reached", "value": 300 }
  ],
  "conditions": [
    { "type": "player_unit_count", "player": "player_2", 
      "operator": "lt", "value": 10 }
  ],
  "actions": [
    { "type": "spawn_units", "unit_id": "orc_warrior", 
      "count": 15, "region": "north_gate", "owner": "player_2" },
    { "type": "display_message", "text": "Reinforcements arrive!", 
      "duration": 5.0 },
    { "type": "play_sound", "sound_id": "war_horn" }
  ]
}
```

The engine validates all triggers at scenario load time: schema validation confirms structural integrity, reference validation ensures all entity IDs, region names, and player references exist in the scenario, logic validation checks numeric ranges, and cycle detection prevents infinite trigger loops.

### Natural language trigger authoring

The LLM integration uses a structured prompt containing the complete trigger schema, the list of all entities/regions/players defined in the current scenario, valid event/condition/action types with their parameters, and one or two examples. The system prompt constrains the LLM to output only valid JSON matching the trigger schema.

The flow: creator types English → HTTP POST to Claude API (or local Ollama fallback) → JSON trigger definition → schema validator → reference resolver (fuzzy-matches entity names against registry) → preview in the Trigger Editor UI → creator reviews, edits if needed, and confirms.

**Risks and mitigations.** Hallucinated entity names: the LLM receives the exact list of valid names and the validator cross-references output against the registry. Invalid logic structures: JSON Schema validation rejects malformed triggers before display. Non-deterministic results: temperature set to 0, response cached for identical prompts. Ambiguity: the system asks clarifying questions via a follow-up prompt. Latency: a loading spinner masks the 2–5 second API call; common patterns are cached locally.

The fallback chain: try Claude API → on failure (timeout, offline, rate limit) → try local Ollama with Llama 3.1 8B → on failure → offer template-based trigger creation from a preset library.

## 5. The Creation Suite

The creation suite is the collection of in-game tools that Architects and Tinkerers use to build scenarios. Its UX is informed by the successes and failures of five landmark editors: the Warcraft III World Editor (accessible, consolidated, beloved), the StarCraft 2 Galaxy Editor (powerful but fatally complex), Halo Forge (immediate 3D manipulation), Dreams (creative but overwhelming), and Super Mario Maker (radical simplicity with a mandatory play-test loop).

### Design principles from editor history

The Warcraft III Object Editor consolidated all unit data — stats, art, sounds, abilities — into a single panel. StarCraft 2 scattered the same data across five separate editors (Unit, Actor, Weapon, Effect, Damage Effect) requiring manual cross-referencing. Community feedback was damning: the SC2 Data Editor was called "the single biggest problem" causing "the most people to just stop and give up." **Chimera follows the WC3 model: one entity, one view, all data consolidated.**

Mario Maker proved that requiring creators to complete their own levels before publishing dramatically increases content quality. **Chimera mandates a play-test-and-win gate before any scenario can be published.**

Dreams demonstrated that hover-over tooltips on every control element reduce learning curve significantly. **Every field, button, and panel in Chimera's creation suite displays a descriptive tooltip on hover.**

Fortnite Creative showed that only ~0.2% of players create content. **The creation suite must be invisible to Commanders — they never see it unless they choose to.** It must also be immediately rewarding for Architects — creating a basic playable scenario should take under 15 minutes.

### Minimum viable creation suite

The MVP creation suite contains six essential tools:

**Terrain Editor.** Brush-based sculpting (raise, lower, smooth, flatten) plus texture painting with 3–4 biome layers. Built on the Terrain3D GDExtension (3,700 GitHub stars, C++ performance, C# accessible), which supports GPU clipmap meshes, up to 32 texture layers, and foliage instancing. Runtime heightmap modification works by writing directly to Terrain3D's Image storage, then triggering a terrain update. Collider regeneration is debounced — updates fire 0.5 seconds after the last brush stroke to prevent frame hitches during active painting.

**Entity Placer.** Drag from a palette panel, place in world. Raycast from mouse position to terrain collision, instantiate a preview ghost, confirm placement on click. Grid snap and free placement modes toggle via hotkey. Placed entities store position, rotation, owner, and type ID in the scenario data.

**Start Position Definer.** Simple flag markers assigned to player numbers. Minimum 2, maximum 8. Each start position includes a configurable starting unit/building list and starting resource amounts.

**Resource Node Placer.** Select from a palette of resource node types (Ore Deposit, Crystal Formation, or creator-defined types). Place on terrain with configurable total supply per node.

**Faction Definer.** A five-step wizard that walks creators through faction creation in 5–12 minutes. Step 1: Name and color (30 seconds). Step 2: Unit roster — drag units from the library into role slots for Worker, Fighter, Ranged, and optional additional slots (2–5 minutes). Step 3: Building roster and tech tree — drag buildings into a dependency chain visual editor (2–5 minutes). Step 4: Starting conditions — resource sliders and a starting unit list (1 minute). Step 5: AI behavior preset — dropdown with Aggressive, Balanced, Defensive, and Passive (30 seconds).

**Win Condition Setter.** Select from the Tier 1 presets or switch to the Trigger Editor for custom logic.

### Entity editor UX

The Unit Card Editor presents all data for a single entity in one consolidated view. The left panel shows a live rotating 3D model preview and buttons to change model/icon. The right panel groups stats into sections: Combat (HP, Attack, Range, Armor, Speed as sliders with numeric input), Economy (costs, build time), and Abilities (pick from a library of pre-built abilities). Every slider shows min/max bounds and accepts both drag and typed values.

A template system accelerates creation: "Start from Footman," "Start from Archer," "Start from Worker" presets populate all fields with reasonable defaults that creators then customize. A comparison view enables side-by-side stat review against existing units for balance checking. A green validation badge appears when a unit is complete (has model, valid stats, and a building that produces it).

### Visual scripting for the Trigger Editor

The advanced trigger mode uses Godot's `GraphEdit` control node — confirmed available at runtime, not editor-only — to present a visual node graph. Pre-built node categories include Events (Timer, Region Enter/Leave, Unit Dies, Building Completed), Conditions (Unit Count, Resource Amount, Boolean Variable), Actions (Spawn, Kill, Move, Message, Sound, Camera, Victory/Defeat), and Flow (Branch, Loop, Sequence, Random Choice).

Both the ECA list view and the visual graph view edit the same underlying trigger data model. Creators switch between views with a toggle button. The ECA list is the default for Architects. The graph view is surfaced when a creator selects "Advanced Mode" or explicitly switches. The system serializes triggers to JSON regardless of which view authored them.

The `GraphEdit` API in Godot 4.4+ is marked "Experimental," meaning the API may change in future engine versions. The implementation wraps all GraphEdit calls in an abstraction layer to insulate against upstream changes.

## 6. Multiplayer architecture

### Network model: server-authoritative deterministic lockstep

Project Chimera uses a **server-authoritative command-based architecture** with deterministic simulation. This combines the bandwidth efficiency of lockstep (only commands traverse the network) with the anti-cheat properties of server authority (the server runs the canonical simulation and filters information per player).

The flow per simulation tick:

1. Each client collects player commands (Move, Attack, Build, etc.) and sends them to the dedicated server. Commands are timestamped with the current tick number.
2. The server validates each command against the authoritative game state: does this player own the commanded units? Does their faction have access to the ordered building? Do they have sufficient resources?
3. The server broadcasts the merged, validated command set for tick N to all clients.
4. All clients and the server advance the deterministic simulation by one tick using the identical command set.
5. Every 60 ticks (~2 seconds), clients compute a state checksum and report it. The server compares checksums to detect desynchronization.

This architecture sends approximately **20 bytes per command**. Eight players issuing ~10 commands per second produce **1.6 KB/s of network traffic** — trivial for any connection. This scales to thousands of units without bandwidth issues, because command count depends on player input rate, not unit count.

### Deterministic simulation requirements

Determinism is the hardest technical challenge in RTS networking. All clients and the server must compute identical simulation results from identical inputs.

**Fixed-point math** replaces all floating-point arithmetic in the simulation layer. IEEE 754 float operations produce different results across CPU architectures (x86 vs ARM), compiler optimizations, and even instruction orderings. The simulation uses a custom `FixedPoint` C# struct (or an existing library like FixedMath.Net from NuGet) with 16.16 fixed-point representation. All positions, velocities, damage calculations, and resource amounts use `FixedPoint` exclusively.

**Deterministic RNG** uses a shared seed distributed at match start. A custom Xorshift PRNG ensures identical random sequences on all clients, called in deterministic order (entities processed by ascending entity ID every tick).

**Ordered execution** processes all entities in the same order on every client. Entities are sorted by unique ID before any per-tick update. No `foreach` over unordered collections. No hash map iteration where order might vary.

**Fixed tick rate** separates simulation from rendering. The simulation advances at exactly **30 ticks per second** regardless of frame rate. The renderer interpolates between simulation states for visual smoothness.

### Input delay and client-side feedback

RTS games use **input delay** rather than prediction-rollback. When a player issues a command at tick T, it is scheduled for execution at tick T+N, where N is the input delay buffer (typically 4–8 simulation ticks, or **133–266ms** at 30 ticks/sec). This delay accommodates network latency: all clients need time to receive the command before the execution tick arrives.

**Immediate feedback masks perceived latency.** When a player clicks to move units, the client instantly plays a selection sound, shows rally point markers, and displays an "order acknowledged" animation. The units themselves don't move until the server-confirmed tick, but the immediate audio-visual response makes the delay feel intentional rather than laggy. RTS players accept this — issuing orders to military units *should* feel like giving commands that take a moment to execute.

The delay adapts to network conditions. If the worst-latency player in the session has 200ms ping, N increases to accommodate. The system displays the current game speed/delay in the UI so players understand the tradeoff.

### Transport layer and server infrastructure

**ENetMultiplayerPeer** provides the transport layer — UDP-based with reliable and unreliable channels, low latency, and built into Godot. WebSocket serves as a fallback for firewalled environments. WebRTC is not used (P2P topology conflicts with server-authoritative design).

The dedicated server exports as a **Linux Docker container** using Godot's headless export mode with resource stripping (no textures, audio, or models on the server). For early development and small player bases, a **$10–20/month VPS** hosts both the game server and the Nakama instance.

### Lobby and matchmaking via Nakama

**Nakama** (Heroic Labs, open-source) handles authentication, matchmaking, lobby state, presence, and chat. It provides an official Godot 4 client via the Asset Library and a C# .NET client via NuGet. Nakama runs in a single Docker container alongside PostgreSQL.

The matchmaking flow: player selects a scenario → Nakama matchmaker groups players with matching scenario ID, player count, and skill range → Nakama relays connection info → clients connect to the dedicated Godot server instance → the game session begins on Godot's ENet networking.

Nakama features used at launch: email and Steam authentication, real-time matchmaking with JSON label queries, match listing and lobby browser, parties (group up before matchmaking), and match chat. Leaderboards and tournaments are post-launch additions.

### Content verification in multiplayer

Before any match starts, every participant must have the identical scenario version. The host selects a scenario; the server records its **mod.io mod ID, file version ID, and SHA-256 content hash**. When each player joins the lobby, their client reports the locally installed version. If hashes mismatch, the player sees "Update Required" with a one-click download button that fetches the correct version from mod.io. The game cannot launch until all hashes match.

This is automated end-to-end. Players never manually manage mod versions. The system draws on lessons from Paradox Games (Stellaris, Crusader Kings III), where checksum mismatches are the number-one multiplayer pain point for modded games.

### Anti-cheat strategy

The primary anti-cheat measure is **server authority itself**. All simulation runs on the server. Clients submit commands and receive only the game state they're entitled to see. Server-side fog of war filtering means clients literally never receive data about enemy units outside their vision — map hacking is impossible because the data doesn't exist on the client.

A creation platform faces unique challenges: custom unit stats aren't "cheats" but intended design. The server validates commands against the loaded scenario definition (not hardcoded values), loads content from the hash-verified mod.io package (not client-supplied data), and rate-limits commands to prevent spam.

For a solo developer, this server-authoritative approach is **Tier 1 anti-cheat** (must-have). Tier 2 additions (replay recording, periodic state checksums, statistics-based anomaly detection) arrive post-launch. Client-side anti-cheat software (kernel-level protections, invasive monitoring) is explicitly out of scope — it's an arms race that requires dedicated security teams.

## 7. UGC pipeline

### mod.io integration specification

**No official mod.io SDK exists for Godot.** The platform provides official plugins only for Unity and Unreal Engine. For Godot C#, the recommended approach is calling the **mod.io REST API directly** using C#'s `HttpClient`. This provides full control, avoids dependency on community plugins that may lag behind API changes, and leverages C#'s strong JSON serialization libraries.

A community GDExtension plugin (aNaOH/modio-godot, Asset Library #2241) exists for basic operations, but for a C# project, native HTTP calls are cleaner and more maintainable.

**Creator upload flow.** The creator builds a scenario in the editor → the editor exports a `.chimera.zip` content package → the creator clicks "Publish" → the client calls `POST /games/{game_id}/mods` to create the listing (name, summary, tags, thumbnail) → calls `POST /games/{game_id}/mods/{mod_id}/files` to upload the package → mod.io runs automated scanning (virus check, file validation, profanity filters) → content goes live or enters moderation queue based on configuration.

**Player browse/install flow.** The player opens the in-game Content Browser → the client calls `GET /games/{game_id}/mods` with filter parameters → results display in a scrollable grid with thumbnails, titles, ratings, and subscriber counts → clicking a scenario shows the detail page → "Subscribe" triggers download and local installation with hash verification → the scenario appears in "My Content."

mod.io provides a **four-layer moderation system**: automated scanning (virus, MIME enforcement, profanity filters including Xbox Live compliance), custom rules engine (flag specific file types, block repeat violators), configurable content approval (pre-publish review or post-publish moderation), and community reporting (player flags route to moderation dashboard). This handles the heavy moderation lifting for a solo developer.

### Content package schema

Each scenario is a self-contained `.chimera.zip` file containing all data needed to play. No cross-package dependencies for V1 — each scenario is fully independent, avoiding the complexity explosion of dependency resolution.

```
scenario.chimera.zip/
├── manifest.json          # Metadata, version, compatibility
├── map/
│   ├── terrain.bin        # Heightmap and texture indices (binary)
│   ├── terrain_meta.json  # Dimensions, tile size, biome config
│   └── props.json         # Decorative prop positions
├── factions/              # One JSON file per faction definition
├── entities/
│   ├── units.json         # All unit type definitions
│   ├── buildings.json     # Building definitions
│   └── projectiles.json   # Projectile definitions
├── tech/
│   └── tech_trees.json    # Per-faction tech trees
├── scenario/
│   ├── setup.json         # Start positions, resources, player config
│   ├── triggers.json      # All scenario triggers and win conditions
│   └── objectives.json    # Player-facing objective descriptions
├── assets/
│   ├── models/            # Custom 3D models (.glb)
│   ├── sprites/           # Custom sprite sheets (.png)
│   ├── audio/             # Sound effects (.ogg)
│   └── portraits/         # Unit/faction portraits (.png)
├── preview/
│   ├── thumbnail.png      # 16:9 thumbnail for listings
│   └── screenshots/       # Up to 5 screenshots
└── checksum.sha256        # Hash of all content files
```

The manifest includes semantic versioning, minimum/maximum Chimera version compatibility, player count range, tags, and the content hash. Unit stats, buildings, tech trees, and triggers use **JSON** — human-readable, easy to diff and debug. Terrain data uses a custom binary format for compactness and fast loading. The trigger system uses a **declarative JSON DSL**, never arbitrary scripts, ensuring the server can validate every trigger against a known set of allowed operations.

### Community features and discovery

**Launch MVP features:** In-game content browser with search, filtering by tags (Game Mode, Player Count, Map Size, Theme, Difficulty), and sorting (Popular, Top Rated, Newest). One-click subscribe and install. Star ratings (1–5). "Play Now" button that auto-creates a lobby. Creator profile pages listing published scenarios and total subscribers. Report button feeding into mod.io moderation.

**Post-launch Phase 1:** Featured/curated section (hand-picked weekly — the single most effective quality signal). "Trending" and "New & Notable" categories. In-game reviews. Version update notifications for subscribed content.

**Post-launch Phase 2:** Creator analytics dashboard (play counts, ratings over time). Collections and playlists. Follow creators. Community creation jams with themes and deadlines.

Quality is surfaced through multiple mechanisms. A **minimum quality gate** requires a thumbnail, description over 100 characters, at least one screenshot, and successful automated validation before publishing. A **weighted rating** algorithm considers number of ratings, completion rate, unique players, and recency — not just average stars. The **proof-of-play gate** from Mario Maker requires creators to successfully play through their own scenario before publishing. Content that receives multiple community reports auto-hides pending review.

**IP ownership** follows the Warcraft III Reforged cautionary tale: Blizzard's claim of ownership over all custom content devastated creator motivation. **Chimera creators own their content entirely.** The platform license grants Chimera a non-exclusive right to host and distribute, nothing more.

## 8. Asset pipeline

### AI 3D model generation: confirmed tools and pipeline

Two tools anchor the 3D pipeline, chosen for quality, license terms, and Godot compatibility:

**Hunyuan3D v2.1** (Tencent, open-source, Apache 2.0 license) is the primary local generation tool. It runs a two-stage pipeline: a Diffusion Transformer generates geometry, then a Paint module applies PBR textures (albedo, normal, roughness, metallic). PBR maps ensure models react properly to Godot's dynamic lighting. It runs locally on **6GB+ VRAM** (RTX 3060 averages ~30 seconds per generation), includes an auto-retopology module (PolyGen), and exports to OBJ, GLB, FBX, and USDZ. A Blender addon enables direct integration. The Apache 2.0 license permits unrestricted commercial use.

**Tripo AI** (tripo3d.ai) serves as the cloud-based complement with the most generous free tier among commercial tools. Version 2.5 (January 2025) introduced Smart Retopology, AI Texturing, one-click rigging, and — critically — a **Godot plugin** for direct export. It supports stylization modes (cartoon, voxel, clay) that align with Chimera's low-poly aesthetic. Commercial use is permitted on paid plans; the free tier is sufficient for prototyping.

**Backup options.** TripoSR (MIT license, sub-second generation, lower quality) for rapid batch props. Meshy.ai (200 free credits/month, CC BY 4.0 on free tier, full commercial rights on $20/month Pro) for its Godot plugin and diverse art style presets. Sloyd.ai for unlimited free procedural generation of simple props with clean topology.

### End-to-end 3D pipeline

The pipeline flows through five stages, all using free tools:

**Stage 1 — Concept generation.** A text prompt or concept sketch (even from Stable Diffusion) feeds into Hunyuan3D (local) or Tripo (cloud). For consistency, all prompts begin with the style prefix: "low-poly stylized RTS game asset, clean geometric shapes, soft pastel colors, slight cel-shading, white background, studio lighting" followed by the specific asset description.

**Stage 2 — Retopology and cleanup.** Raw AI output typically produces 10K–50K triangles. Tripo's Smart Retopology or Blender's Instant Meshes addon reduces this to target budgets. Target polygon counts for RTS: **1,000–3,000 triangles** for units (LOD0), **3,000–10,000** for buildings, **200–1,000** for props.

**Stage 3 — UV unwrapping and texturing.** Hunyuan3D's SeamGPT handles automatic UV unwrapping. PBR textures from the generation stage are reviewed and touched up in Blender's texture painting mode if needed.

**Stage 4 — LOD generation and collision.** Blender's Decimate modifier produces 3 LOD levels per asset (LOD0 at full detail, LOD1 at ~40%, LOD2 at ~15%). A simplified convex hull generates the collision shape at less than 1% of visual mesh triangles.

**Stage 5 — Godot import.** Export as GLB (Godot's preferred format). Import auto-generates MeshInstance3D. Configure LOD distances via MeshInstance3D settings. Attach CollisionShape3D with the simplified mesh. Set up materials using StandardMaterial3D for PBR.

### AI 2D asset pipeline

**Stable Diffusion 3.5** (or FLUX.1) via ComfyUI generates UI icons, unit portraits, faction banners, and textures. A LoRA trained on 20–50 reference images in the target style (using the free Kohya_ss trainer, ~30 minutes on a consumer GPU) ensures visual consistency across all 2D assets.

Godot UI resolution standards: icons at 64×64 or 128×128 PNG with alpha, portraits at 256×256 or 512×512, UI panels as 9-patch PNGs for scalability, textures at power-of-two resolutions (256, 512, 1024). All images export as PNG or WebP.

### Art style consistency strategy

Consistency across hundreds of AI-generated assets is the pipeline's hardest challenge. Five mechanisms enforce it:

1. **Universal style prefix** in all generation prompts, defining the low-poly aesthetic, color palette, and lighting.
2. **Negative prompt blacklist** applied to every generation: "photorealistic, high detail, organic textures, noisy, dark lighting, perspective distortion, watermark."
3. **LoRA model** for 2D assets ensuring consistent rendering style.
4. **Shared material library** in Godot — a set of StandardMaterial3D presets with consistent roughness, metallic values, and color palettes that all assets use.
5. **Post-processing shader** applied globally in Godot — a subtle cel-shading outline and color correction pass that unifies assets that might otherwise look inconsistent.

## 9. AI integration

### LLM service architecture

A central `LLMService` class in C# manages all AI interactions, providing a unified interface to both cloud (Claude API) and local (Ollama) backends. The fallback chain: attempt Claude API → on failure → attempt local Ollama → on failure → return template/cached response.

**Cloud integration** uses the official Anthropic C# SDK (NuGet package `Anthropic`, version 10+), which reads the API key from environment variables, supports async/await natively in Godot's frame loop, and provides structured output via tool use. For scenarios where the NuGet dependency is undesirable, Godot's `HttpRequest` node or C#'s `HttpClient` call the REST API directly.

**Local integration** connects to Ollama running as an HTTP server on `localhost:11434`. The recommended local model is **Llama 3.1 8B** (Q4_K_M quantization, ~4.5GB, runs on 6GB VRAM at 40+ tokens/second). For structured JSON output, Ollama natively supports JSON Schema enforcement via the `format` parameter — the request includes the exact schema and the model is constrained to produce only valid JSON matching that schema. Alternative local models: Qwen3 8B for superior structured output, Mistral 7B for creative text generation, Phi-3 Mini 3.8B for fast inference on lower-end hardware.

Ollama configuration for game use: `OLLAMA_NUM_PARALLEL=1`, `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_FLASH_ATTENTION=1`, `OLLAMA_KV_CACHE_TYPE=q8_0`. This minimizes VRAM usage and prevents resource contention with the running game.

### AI features by player type

**For Commanders.** AI opponent behavior uses a **utility AI system** — no machine learning, only weighted decision trees that evaluate game state and select the highest-scoring action. Each possible action (build workers, expand, attack, research, defend) has a scoring function based on current resources, army composition, scouting information, and time. Difficulty levels adjust weights and reaction delays. The system tracks player patterns using simple counters (rush frequency, turtle frequency) and weights counter-strategies accordingly. This is fully deterministic — essential for multiplayer synchronization.

**For Architects.** Natural language trigger scripting (detailed in Section 4). AI-generated unit and faction names/lore via local LLM with constrained JSON output. AI-assisted map generation: procedural noise terrain as a base → rule-based feature placement (resources, choke points, symmetry for competitive maps) → optional LLM refinement pass suggesting improvements for balance and flow. AI balance analysis: serialize unit stats to JSON, prompt the LLM to identify cost-effectiveness outliers and suggest adjustments.

**For Tinkerers.** Full API access to the LLMService for custom tool building. Ollama local integration for offline modding sessions. AI-assisted code generation via Claude Code for writing custom game logic. Debug overlays showing AI decision weights in real time.

### Natural language trigger scripting — technical specification

The prompt sent to the LLM includes: the complete JSON schema for valid triggers, a registry of all entities/regions/players in the current scenario (exact IDs), the complete list of valid event types, condition types, and action types with their parameters, and 2–3 examples of natural language → trigger JSON conversions.

The response parser in C# deserializes the JSON, then runs five validation passes: schema validation (does the JSON match `TriggerDefinition`?), entity reference validation (does every referenced entity ID exist in the scenario registry?), operator validation (is the operator compatible with the condition type?), range validation (are numeric values within acceptable bounds?), and safety validation (are spawn counts capped? are there potential infinite loops with other triggers?).

Fuzzy matching handles near-miss entity references: if the LLM outputs "barrack" but the registry contains "barracks_t1," the validator suggests "Did you mean 'barracks_t1'?" with a one-click accept. Error messages are displayed inline in the Trigger Editor UI with clear, human-readable explanations and "Fix" buttons.

## 10. Phased build roadmap

The roadmap spans **19–31 months** across six phases. Each phase ends with a playable, demonstrable build. The critical insight from every successful solo-dev strategy game (Dwarf Fortress, RimWorld, Factorio, Mindustry) is the same: **ship small and grow over years.** Stormgate's failure — launching an RTS creation platform with competitive PvP, co-op, campaign, three factions, a map editor, and F2P monetization simultaneously — is the cautionary counterexample.

### Phase 0 — Technical prototypes and risk reduction (months 1–3)

**Objective:** Validate or kill the three highest-risk technical bets before committing to full development.

**Deliverables:** (1) Unit count stress test — 500+ units with pathfinding and combat at 60 FPS using MultiMesh rendering and C# SoA simulation. (2) Basic edit-play loop — place terrain, drop units, press Play, watch them fight, press Edit, resume placing. (3) AI art pipeline evaluation — generate one complete faction's visual assets (8 unit types, 4 buildings) and assess consistency.

**Success criteria:** If the unit stress test fails at 500 units after optimization, evaluate whether GDExtension (C++) can solve it or whether Godot is the wrong engine. If the edit-play loop feels sluggish or broken, investigate architecture changes. If AI assets look incoherent across a faction, commit to a simpler art style (pixel art, voxel) or plan for more manual cleanup time.

**Playable at end:** A technical demo with moving, fighting units on test terrain, plus a primitive editor toggling between build and play modes.

### Phase 1 — Core RTS engine, single-player (months 3–9)

**Objective:** Build a compelling single-player RTS that is fun on its own merits, independent of any creation tools.

**Deliverables:** One complete faction with **8–12 units and 4–6 buildings**. Resource gathering with two resource types. Base building with construction mechanics. Combat system with damage/armor matrix. AI opponent with finite state machine behavior at 2 difficulty levels. Fog of war. 3–5 hand-crafted skirmish maps. A guided tutorial campaign of 5–8 missions. UI for selection, commands, build menus, minimap. Save/load game state.

**Success criteria:** External playtesters find the core loop fun in blind tests. Average session length exceeds 30 minutes. The question "when can I play more?" emerges organically.

**Risk checkpoint:** If playtesting reveals the core RTS loop is not fun, **stop everything and iterate on game design** before adding creation tools. A creation platform built on an unfun game is worthless. The RTS must be compelling standalone — the lesson from Dreams' failure is that great tools without great gameplay attract neither creators nor players.

**Playable at end:** A small but complete single-player RTS game.

### Phase 2 — Creation suite basics and Early Access launch (months 9–14)

**Objective:** Ship the creation suite alongside the base game as a Steam Early Access release.

**Deliverables:** Terrain editor (sculpt, paint, place props). Entity placement mode. Tier 2 trigger editor (ECA pattern). Custom scenario save/load. Basic data modding (unit stats, costs via JSON). Instant play-test toggle. Undo/redo. A second faction for asymmetric gameplay. 5+ additional skirmish maps. **Steam Early Access launch.**

**Success criteria:** Players create and share maps via Discord file sharing (formal UGC pipeline comes later). At least 50 player-created maps within the first month. Steam Early Access reviews above 70% positive.

**Business model:** Early Access price of **$15–25**. No microtransactions, no free-to-play. One-time purchase funds continued development. Stormgate's F2P model confused the RTS audience and required massive player volume that never materialized. A premium price signals confidence and sets clear expectations.

**Playable at end:** A purchasable RTS with built-in creation tools — the minimum viable creation platform.

### Phase 3 — Multiplayer (months 14–20)

**Objective:** Add networked play for 2–8 players with full anti-cheat.

**Deliverables:** Deterministic simulation layer with fixed-point math. Server-authoritative command-based networking via ENet. Dedicated server builds (Linux Docker). Nakama integration for lobby, matchmaking, and authentication. 1v1 and 2v2 support. Spectator mode. Replay system (nearly free with deterministic simulation — record the command stream, replay it). Basic anti-cheat (server fog of war, command validation, content hashing). 3–5 competitive maps balanced for multiplayer.

**Success criteria:** Two players complete a full match with less than 50ms perceived latency and zero desyncs in 95%+ of games.

**Risk checkpoint:** If deterministic simulation proves intractable in Godot (desync rates too high despite fixed-point math), evaluate a server-authoritative state-snapshot architecture with delta compression as the fallback. If the player base is too small for matchmaking queues, focus on "play with friends" and enhance the AI opponent instead.

### Phase 4 — UGC pipeline and community (months 20–25)

**Objective:** Formalize content sharing with mod.io integration and in-game discovery.

**Deliverables:** mod.io REST API integration for upload, browse, subscribe, rate, and download. In-game content browser with search, filters, and sorting. Content package format (.chimera.zip) with automated validation. Creator profiles. Community reporting. Featured/curated section. Tier 3 visual scripting (GraphEdit-based trigger node graph). Custom unit/building creation tools in the editor. A third faction. Campaign expansion content.

**Success criteria:** 500+ workshop items within 3 months. An active creator community on Discord. At least 10 "high-quality" community scenarios or campaign packs.

### Phase 5 — Polish, AI features, and 1.0 release (months 25–31)

**Objective:** Ship the polished 1.0 product with AI-powered creation accelerators.

**Deliverables:** Natural language trigger scripting (LLM integration). AI-assisted map generation. AI balance analysis tools. Performance optimization pass. Accessibility features. Linux build support. Advanced editor features (particle effects, sound triggers). Full 1.0 release. Post-launch: seasonal featured content rotations, community creation jams.

**Success criteria:** Overall Steam reviews above 80% positive. Stable concurrent player base. Active creation community. Revenue sustaining continued solo development.

### Risk-ordered prototype queue

The prototyping order prioritizes the riskiest bets that could invalidate the entire project:

1. **Unit count at scale** (weeks 1–3) — the project-killing risk. If Godot 4 + C# cannot handle 500 units at 60 FPS, nothing else matters.
2. **Edit-play loop** (weeks 3–5) — the platform-defining feature. If switching between creation and play feels broken, the creation platform concept fails.
3. **AI art pipeline** (weeks 5–7) — the solo-dev enabler. If AI cannot produce a visually coherent faction of assets, the art pipeline requires fundamental redesign.
4. **Deterministic multiplayer** (Phase 3 start) — deferred because single-player must prove fun first, but prototyped at the beginning of Phase 3 before committing months to networking code.
5. **UGC packaging** (Phase 4 start) — lowest risk, as mod.io is a known-solved platform. Prototyped when relevant.

## 11. Open questions and prototypes needed

### Unresolved technical questions

**Godot 4 C# fixed-point math performance.** The deterministic simulation requires all arithmetic in fixed-point. C#'s `decimal` type is too slow; a custom struct is needed. The question: can a C# fixed-point library achieve acceptable performance for 1,000+ entity updates per tick at 30 Hz, or will this require GDExtension (C++)? This is a Phase 3 prototype.

**Flow field implementation at scale.** No production-ready Godot flow field pathfinding exists. Community implementations (GodotFlowField) are early-stage with known performance issues. The custom C# implementation must be profiled against the navigation server approach. Will flow fields outperform rate-limited A* at 500+ units, or do they introduce their own scaling problems?

**Terrain3D runtime modification stability.** Terrain3D is a GDExtension designed primarily for editor use. Runtime heightmap modification (for the creation suite) has been validated in principle but not stress-tested with rapid brush strokes, large maps, and simultaneous collider updates. Extended testing is needed.

**GraphEdit API stability.** Godot marks GraphEdit as "Experimental" in 4.4+. The API may change in 4.7 or later. The abstraction layer mitigates this, but a breaking change could require significant rework of the visual scripting interface.

**NativeAOT server builds with Godot C#.** Self-contained server deployment via NativeAOT avoids requiring .NET runtime on server machines, but the combination of Godot + C# + NativeAOT + headless mode has limited community testing. This needs early validation.

### Design questions requiring playtesting

**Optimal complexity ceiling for the Trigger Editor.** The Warcraft III trigger editor succeeded because it was simple enough for beginners yet powerful enough for the top 80% of custom game scenarios. StarCraft 2's Galaxy Editor failed because it was too powerful and too complex. Where exactly does the Trigger Editor's vocabulary stop? How many event types, condition types, and action types constitute the sweet spot? This requires iterative playtesting with real Architects.

**AI opponent quality floor.** How good must the AI be for single-player to feel satisfying? Utility AI with hand-tuned weights can produce competent play, but RTS players expect the AI to adapt to their strategy. The question: does a pattern-tracking counter-strategy system (tracking rush frequency, turtle frequency) produce noticeably better AI, or is static utility scoring sufficient for Phase 1?

**Creation suite onboarding flow.** The five-step faction wizard targets a 5–12 minute creation time. Is this fast enough to capture Architects' interest, or does the first session need to produce a playable result in under 5 minutes (à la Mario Maker's radical simplicity)? Should the creation suite open with a guided "Your First Scenario in 10 Minutes" tutorial, or with a template gallery?

### Prototypes to build before full commitment

- **500-unit combat stress test** with MultiMesh rendering, C# SoA simulation, and rate-limited pathfinding (Phase 0, weeks 1–3)
- **Edit-play mode toggle** with terrain painting, entity placement, and seamless transition (Phase 0, weeks 3–5)
- **Complete AI-generated faction** — 8 unit models, 4 building models, portraits, icons — assessed for visual consistency (Phase 0, weeks 5–7)
- **Fixed-point deterministic simulation** running identically on two local instances with checksum comparison (Phase 3, week 1–2)
- **mod.io REST API round-trip** — upload a test package from Godot C#, browse it in-game, download and load it (Phase 4, week 1)

### What this project is not

Project Chimera is not a competitor to Roblox or Fortnite Creative in platform scale. Those products have hundreds to thousands of engineers and billions of dollars in infrastructure. Chimera is a **focused, genre-specific creation platform** for RTS enthusiasts, modeled on the Warcraft III World Editor's scope and ambition. The goal is not to build the universal game-creation platform but to ship the definitive RTS-specific one — the tool that could birth the next DotA within a single, well-designed system built by one developer with AI as a force multiplier.

The realistic scope: a solid single-player RTS with 1–2 factions, a functional creation suite with local save/load, mod.io integration for sharing, basic multiplayer, and an AI-assisted art pipeline. Three asymmetric factions, advanced visual scripting, AI asset generation exposed to end users, and a creator marketplace are stretch goals achievable only if the core product succeeds. The minimum viable path mirrors Mindustry: start as a game jam prototype, release early, iterate based on community feedback, and let the modding community extend the game's lifespan beyond what a solo developer could achieve alone.

**The RTS must be fun standalone before the creation platform matters.** Launch as a game. Grow into a platform.
